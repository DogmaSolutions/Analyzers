# DSA030

Entity Framework queries should explicitly specify a change tracking strategy.

| Property     | Value                                                                                          |
|:-------------|:-----------------------------------------------------------------------------------------------|
| Rule ID      | DSA030                                                                                        |
| Title        | Entity Framework queries should explicitly specify a change tracking strategy                  |
| Category     | Best Practice                                                                                 |
| Severity     | 💡 Suggestion                                                                                  |
| Refactoring available | ✅                                                                                             |
| Code-review tools available | ✅                                                                                             |
| Is enabled   | ✅                                                                                             |

## 💡 Rationale

This rule fires when an Entity Framework Core LINQ query is materialized (via `ToListAsync`, `FirstOrDefaultAsync`, `CountAsync`, `ToList`, and similar terminal operations) without a preceding call to `AsNoTracking()`, `AsTracking()`, or `AsNoTrackingWithIdentityResolution()` anywhere in the query chain. The analyzer also traces queries stored in local variables and received as `IQueryable<T>` parameters, performing a best-effort cross-method analysis to determine whether callers have already specified a tracking strategy before passing the query.

**What the tracking methods do**: Entity Framework Core's change tracker monitors every entity returned by a query so that `SaveChanges()` can detect modifications and persist them. This tracking has a per-entity cost in memory and CPU. For read-only scenarios — reports, projections, API responses — this overhead is entirely wasted. Conversely, for write scenarios, tracking is essential and should be explicitly requested rather than silently inherited from the default configuration. An explicit tracking choice at the query site makes the developer's intent clear, prevents subtle bugs when the global default changes, and avoids unnecessary performance overhead.

## 🔐 Security implications

### CWE-400: Uncontrolled Resource Consumption (Denial of Service)

Unnecessary change tracking is a **direct amplifier for memory-exhaustion denial-of-service attacks** (CWE-400). The change tracker maintains, for every materialized entity:

1. A reference to the entity instance itself.
2. A **full snapshot copy** of every property value at the time of materialization (used for dirty checking).
3. Internal bookkeeping structures: state entries, relationship fixup maps, and identity-resolution dictionaries.

This means that a tracked query returning _N_ entities with _P_ properties allocates roughly **2N object graphs** plus **N × P** snapshot values. For a read-only endpoint that returns 10,000 rows, the change tracker silently doubles the working-set footprint and adds thousands of short-lived allocations that increase GC Gen0/Gen1 pressure.

An attacker who can influence query parameters — pagination size, filter predicates, sort order, search terms — can craft requests that maximize the number of returned entities. Without `AsNoTracking()`, each such request amplifies memory consumption by the tracking overhead factor. Under sustained load, this amplification exhausts the application's memory budget, triggers aggressive garbage collection pauses, and can cascade into `OutOfMemoryException` crashes or container OOM kills — achieving denial of service without any malformed input that would be caught by input validation.

**Mitigation**: `AsNoTracking()` on read-only queries eliminates the snapshot and bookkeeping allocations entirely, cutting per-entity memory overhead roughly in half and removing the amplification vector.

### CWE-284: Improper Access Control (Stale-State Manipulation)

When entities are tracked by default but never intended for modification, a subtle attack surface emerges. If application code inadvertently exposes a tracked entity through a shared context (e.g., a scoped `DbContext` reused across middleware or service calls within the same request), a downstream component can mutate properties on the tracked entity and call `SaveChanges()` to persist unauthorized modifications. With `AsNoTracking()`, returned entities are **detached from the context**: mutations have no effect on `SaveChanges()`, closing this lateral privilege-escalation path.

### IEC 62443 Certification

In the context of **IEC 62443 certification**, explicit tracking strategy supports multiple foundational requirements:

- **FR 3 — System Integrity**: making the tracking decision explicit at each query site prevents silent behavioral changes when the global `QueryTrackingBehavior` configuration is modified, ensuring that the data access layer behaves **deterministically and as designed**. A change to the global default (e.g., from `TrackAll` to `NoTracking` during a "performance optimization") could silently break write paths that relied on implicit tracking — an integrity violation that is invisible until data loss occurs.
- **FR 7 — Resource Availability**: eliminating unnecessary change tracking on read-only queries reduces memory consumption and GC pressure, improving the application's **resilience to load spikes and resource-exhaustion scenarios**. Systems that must meet SL-2 or higher availability targets cannot afford silent 2x memory amplification on every read query.
- **FR 2 — Use Control**: explicit tracking intent at the query site creates a clear **audit trail in the source code** of which queries are expected to produce modifiable entities and which are not. Code reviewers, static analysis tools, and compliance auditors can verify that write-capable tracking is only used where modifications are intended, supporting the principle of least privilege at the data access layer.

## ⚡ Performance implications

### Memory overhead per tracked entity

The change tracker's per-entity cost is significant and non-obvious:

| Component | Tracked query | `AsNoTracking()` query |
|-----------|--------------|----------------------|
| Entity instance | ✅ allocated | ✅ allocated |
| Original-values snapshot | ✅ allocated (full property copy) | ❌ eliminated |
| `InternalEntityEntry` | ✅ allocated | ❌ eliminated |
| Identity-resolution dictionary entry | ✅ allocated | ❌ eliminated |
| Relationship fixup overhead | ✅ computed | ❌ eliminated |

For a typical entity with 10–20 properties, the snapshot copy alone can equal or exceed the entity's own memory footprint. On queries returning hundreds or thousands of entities, the difference between tracked and untracked materialization is measured in **megabytes of heap pressure per request**.

### GC pressure and latency

Every snapshot and bookkeeping structure is a short-lived allocation that survives at least until the `DbContext` is disposed (typically at request end). In high-throughput services, this creates sustained Gen0/Gen1 GC pressure, increasing pause frequency and tail latency (P99/P999). `AsNoTracking()` eliminates these allocations entirely, resulting in measurably lower GC pause time under load.

### `DetectChanges()` cost

`SaveChanges()` calls `DetectChanges()`, which iterates every tracked entity and compares current property values against the snapshot. Entities from read-only queries that were tracked by default add to this iteration cost without contributing any meaningful state changes — pure overhead on every save operation within the same `DbContext` scope.

### Benchmarked impact

Microsoft's own documentation and community benchmarks consistently show that `AsNoTracking()` queries execute **30–50% faster** for typical read-only workloads due to eliminated snapshot allocation, reduced GC pressure, and skipped identity-resolution lookups. For large result sets, the difference can exceed **2x** in both throughput and memory.

## 🔦 Code sample

```csharp
public class UserService
{
    private readonly MyDbContext _context;

    public UserService(MyDbContext context) { _context = context; }

    public async Task<List<User>> GetActiveUsers_NotOk()
    {
        // this WILL trigger the rule — no tracking strategy specified
        return await _context.Users
            .Where(u => u.IsActive)
            .OrderBy(u => u.Name)
            .ToListAsync();
    }

    public async Task<List<User>> GetActiveUsers_Ok_NoTracking()
    {
        // this WILL NOT trigger the rule — AsNoTracking explicitly specified
        return await _context.Users
            .AsNoTracking()
            .Where(u => u.IsActive)
            .OrderBy(u => u.Name)
            .ToListAsync();
    }

    public async Task<List<User>> GetActiveUsers_Ok_Tracking()
    {
        // this WILL NOT trigger the rule — AsTracking explicitly specified
        return await _context.Users
            .AsTracking()
            .Where(u => u.IsActive)
            .OrderBy(u => u.Name)
            .ToListAsync();
    }
}
```
## ℹ️ Matched patterns (code is NOT OK)

```cs
// All of these are flagged:
_context.Users.ToListAsync()
_context.Users.Where(u => u.IsActive).ToListAsync()
_context.Users.Where(u => u.IsActive).OrderBy(u => u.Name).FirstOrDefaultAsync()
_context.Users.TagWith("query").ToListAsync()
_context.Users.Where(u => u.IsActive).ToList()
_context.Users.CountAsync()
_context.Users.AnyAsync()

// Variable without tracking:
var query = _context.Users.Where(u => u.IsActive);
await query.ToListAsync();  // flagged
```

## ✅ Not matched patterns (code is OK)

```cs
// AsNoTracking specified:
_context.Users.AsNoTracking().Where(u => u.IsActive).ToListAsync()  // not flagged

// AsTracking specified:
_context.Users.AsTracking().Where(u => u.IsActive).ToListAsync()  // not flagged

// AsNoTrackingWithIdentityResolution specified:
_context.Users.AsNoTrackingWithIdentityResolution().ToListAsync()  // not flagged

// Tracking in local variable initializer:
var query = _context.Users.AsNoTracking().Where(u => u.IsActive);
await query.ToListAsync();  // not flagged

// Bulk operations (bypass change tracking by design):
_context.Users.Where(u => !u.IsActive).ExecuteDeleteAsync()  // not flagged
_context.Users.ExecuteUpdateAsync(u => u)  // not flagged

// In-memory collection (not Entity Framework):
users.Where(u => u.IsActive).ToList()  // not flagged

// Subquery inside a Where/Select lambda:
_context.Users.Where(u => _context.Users.Any()).ToListAsync()  // only outer flagged
```

## 🛠️ Included Code Fixer and Mitigation

Two code fixes are available:

**Add .AsNoTracking()**: Inserts `.AsNoTracking()` immediately before the terminal materialization method. This is the recommended fix for read-only queries, as it eliminates change tracking overhead entirely and closes the stale-state manipulation attack surface.

```cs
// Before:
await _context.Users.Where(u => u.IsActive).ToListAsync();

// After:
await _context.Users.Where(u => u.IsActive).AsNoTracking().ToListAsync();
```

**Add .AsTracking()**: Inserts `.AsTracking()` immediately before the terminal materialization method. Use this when the entities will be modified and saved, making the tracking intent explicit and resilient to global configuration changes.

```cs
// Before:
await _context.Users.Where(u => u.IsActive).ToListAsync();

// After:
await _context.Users.Where(u => u.IsActive).AsTracking().ToListAsync();
```
