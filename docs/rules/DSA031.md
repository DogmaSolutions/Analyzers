# DSA031

Use AsNoTracking for Entity Framework queries that do not require change tracking.

| Property     | Value                                                                                          |
|:-------------|:-----------------------------------------------------------------------------------------------|
| Rule ID      | DSA031                                                                                        |
| Title        | Use AsNoTracking for Entity Framework queries that do not require change tracking              |
| Category     | Performance                                                                                   |
| Severity     | ⚠ Warning                                                                                     |
| Code fix     | ✅                                                                                             |
| Is enabled   | ✅                                                                                             |

## Description

This rule fires when an Entity Framework Core query materializes entities without `AsNoTracking()` and the analyzer can **prove** that change tracking is unnecessary. Unlike DSA030 (which suggests specifying *any* tracking strategy), this rule specifically requires `AsNoTracking()` because it detects patterns where tracking is provably wasted.

The analyzer targets two zero-false-positive patterns:

### Pattern A: Projection to non-entity types

When the query chain projects to a non-entity type (anonymous type, value tuple, or primitive) via `.Select()`, the materialized result is never tracked by Entity Framework. Adding `AsNoTracking()` documents read-only intent and protects against regressions if the projection is removed during refactoring.

### Pattern B: Read-only method bodies

When the containing method body satisfies **all** of the following conditions, the materialized entities are provably not mutated:

1. **No `SaveChanges()` / `SaveChangesAsync()`** — the method never persists changes
2. **No DbContext mutation APIs** — no calls to `Add`, `Update`, `Remove`, `Attach`, `Entry`, or their `*Range`/`*Async` variants
3. **No entity property assignments** — no `entity.Property = value` patterns on the entity type
4. **Return type does not involve the entity type** — the method doesn't return entities to callers who might mutate them
5. **Entities are not stored in fields or properties** — they don't escape to class-level state
6. **Entities are not passed as arguments to other methods** — they don't escape to code the analyzer can't inspect

## Security implications

### CWE-400: Uncontrolled Resource Consumption (Denial of Service)

Unnecessary change tracking is a **direct memory amplifier** for denial-of-service attacks. The change tracker maintains, for every tracked entity:

- A reference to the entity instance itself
- A **full snapshot copy** of every property value (for dirty checking)
- Internal bookkeeping: `InternalEntityEntry`, identity-resolution dictionary entries, relationship fixup maps

For a query returning _N_ entities with _P_ properties, tracking roughly **doubles** the memory footprint compared to `AsNoTracking()`. An attacker who can influence query parameters (pagination size, filter predicates, search terms) can craft requests that maximize entity count, amplifying memory consumption through the tracking overhead. Under sustained load, this causes `OutOfMemoryException` crashes or container OOM kills — achieving denial of service without malformed input.

**Mitigation**: `AsNoTracking()` eliminates snapshot and bookkeeping allocations, cutting per-entity memory overhead roughly in half and removing the amplification vector.

### CWE-284: Improper Access Control (Stale-State Manipulation)

When entities are tracked by default but never intended for modification, a subtle attack surface emerges. If tracked entities reach code that shares the same `DbContext` scope (middleware, other services in the same request), that code can mutate properties and call `SaveChanges()` to persist unauthorized changes. With `AsNoTracking()`, returned entities are **detached from the context**: mutations have no effect on `SaveChanges()`, closing this lateral privilege-escalation path.

### IEC 62443 Certification

- **FR 3 — System Integrity**: explicit `AsNoTracking()` on read-only queries ensures deterministic behavior independent of the global `QueryTrackingBehavior` setting. A change to the global default cannot silently introduce tracking overhead or enable unintended persistence.
- **FR 7 — Resource Availability**: eliminating unnecessary tracking reduces memory consumption and GC pressure, improving resilience to load spikes and resource-exhaustion scenarios required for SL-2+ availability targets.

## Performance implications

### Memory overhead per tracked entity

| Component | Tracked query | `AsNoTracking()` query |
|-----------|--------------|----------------------|
| Entity instance | ✅ allocated | ✅ allocated |
| Original-values snapshot | ✅ allocated (full property copy) | ❌ eliminated |
| `InternalEntityEntry` | ✅ allocated | ❌ eliminated |
| Identity-resolution dictionary entry | ✅ allocated | ❌ eliminated |
| Relationship fixup overhead | ✅ computed | ❌ eliminated |

For a typical entity with 10–20 properties, the snapshot copy alone can equal or exceed the entity's own memory footprint. On queries returning hundreds or thousands of entities, the difference is measured in **megabytes of heap pressure per request**.

### GC pressure and latency

Snapshot and bookkeeping structures are short-lived allocations that survive until `DbContext` disposal (typically request end). In high-throughput services, this creates sustained Gen0/Gen1 GC pressure, increasing pause frequency and tail latency (P99/P999).

### `DetectChanges()` cost

`SaveChanges()` iterates every tracked entity and compares current values against snapshots. Entities from read-only queries that were tracked by default add to this iteration — pure overhead.

### Benchmarked impact

Microsoft's documentation and community benchmarks show `AsNoTracking()` queries execute **30–50% faster** for typical read-only workloads. For large result sets, the difference can exceed **2x** in throughput and memory.

## Matched patterns

```cs
// Projection to anonymous type:
_context.Users.Select(u => new { u.Name }).ToListAsync()  // flagged

// Projection to primitive:
_context.Users.Select(u => u.Name).ToListAsync()  // flagged

// Read-only method body (no SaveChanges, no mutations, result doesn't escape):
public async Task LogCount()
{
    var users = await _context.Users.ToListAsync();  // flagged
    Console.WriteLine(users.Count);
}

// Read-only with filter and derived return:
public async Task<int> GetActiveCount()
{
    var users = await _context.Users.Where(u => u.IsActive).ToListAsync();  // flagged
    return users.Count;
}

// Read-only with FirstOrDefault and property read:
public async Task<string> GetFirstName()
{
    var user = await _context.Users.FirstOrDefaultAsync();  // flagged
    return user?.Name;
}
```

## Not matched patterns

```cs
// Tracking already specified:
_context.Users.AsNoTracking().ToListAsync()  // not flagged
_context.Users.AsTracking().ToListAsync()  // not flagged

// Method calls SaveChanges — entities might be persisted:
public async Task UpdateAll() {
    var users = await _context.Users.ToListAsync();  // not flagged
    await _context.SaveChangesAsync();
}

// Entity property assigned — entities are mutated:
public async Task Deactivate() {
    var user = await _context.Users.FirstAsync();  // not flagged
    user.IsActive = false;
}

// Result returned as entity type — caller could mutate:
public async Task<List<User>> GetAll() {
    return await _context.Users.ToListAsync();  // not flagged
}

// Result stored in field — could be mutated later:
public async Task Cache() {
    _cachedUsers = await _context.Users.ToListAsync();  // not flagged
}

// Result passed to method — method could mutate:
public async Task Process() {
    var users = await _context.Users.ToListAsync();  // not flagged
    ProcessUsers(users);
}

// Bulk operations (bypass tracking by design):
_context.Users.ExecuteDeleteAsync()  // not flagged

// In-memory collection (not EF):
new List<User>().Where(u => u.IsActive).ToList()  // not flagged

// Scalar aggregations (not entity-materializing):
_context.Users.CountAsync()  // not flagged (handled by DSA030 instead)
```

## Code fix

**Add .AsNoTracking()**: Inserts `.AsNoTracking()` immediately before the terminal materialization method.

```cs
// Before:
var users = await _context.Users.Where(u => u.IsActive).ToListAsync();

// After:
var users = await _context.Users.Where(u => u.IsActive).AsNoTracking().ToListAsync();
```

## Code sample

```csharp
public class UserService
{
    private readonly MyDbContext _context;

    public UserService(MyDbContext context) { _context = context; }

    public async Task<int> GetActiveCount_NotOk()
    {
        // this WILL trigger the rule — entities are loaded with tracking
        // but the method only reads a count
        var users = await _context.Users
            .Where(u => u.IsActive)
            .ToListAsync();
        return users.Count;
    }

    public async Task<int> GetActiveCount_Ok()
    {
        // this WILL NOT trigger the rule — AsNoTracking eliminates overhead
        var users = await _context.Users
            .AsNoTracking()
            .Where(u => u.IsActive)
            .ToListAsync();
        return users.Count;
    }

    public async Task<List<string>> GetNames_NotOk()
    {
        // this WILL trigger the rule — projection to string means
        // no entities are tracked, but AsNoTracking documents intent
        return await _context.Users
            .Select(u => u.Name)
            .ToListAsync();
    }

    public async Task<List<string>> GetNames_Ok()
    {
        // this WILL NOT trigger the rule
        return await _context.Users
            .AsNoTracking()
            .Select(u => u.Name)
            .ToListAsync();
    }
}
```
