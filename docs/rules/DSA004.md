Use `DateTime.UtcNow` instead of `DateTime.Now`.

## 💡 Rationale

Using `DateTime.Now` into business logics potentially leads to many different problems:

- Incoherence between nodes or processes running in different timezones (even in the same country, i.e. USA, Russia, China, etc)
- Unexpected behaviours in-between legal time changes
- Code conversion problems and loss of timezone info when saving/loading data to/from a datastore

## 🔐 Security implications

### CWE-361: Time and State (7PK)

`DateTime.Now` returns the machine's local time, which is inherently non-portable across time zones and ambiguous during DST transitions. In security-sensitive contexts, this creates concrete risks:

- **Audit trails**: timestamps recorded in local time cannot be reliably correlated across nodes in different time zones. During forensic analysis, investigators must guess which time zone each log entry was recorded in — a problem that compounds in geographically distributed systems.
- **Time-bound access control**: token expiration, session timeouts, and scheduled access windows that rely on local time can be inadvertently extended or shortened by DST changes, creating windows of unauthorized access or unexpected denial of service.
- **Event ordering**: when two events occur close together on nodes in different time zones (or the same node across a DST boundary), local timestamps can place them in the wrong order, obscuring causal chains during incident response.

**IEC 62443 mapping**: FR 3 (System Integrity) and FR 6 (Timely Response to Events) — consistent UTC timestamps are essential for reliable audit trails, event correlation across distributed nodes, and forensic analysis during incident response.

## 👉️ Rules of thumb for prevention and proactive posture
Don't use `DateTime.Now`. Use `DateTime.UtcNow` instead.
