# DSA029

The [Required Attribute](https://learn.microsoft.com/en-us/dotnet/api/system.componentmodel.dataannotations.requiredattribute) has no impact on a
not-nullable [value type](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/builtin-types/value-types) property.

| Property     | Value                                                                                          |
|:-------------|:-----------------------------------------------------------------------------------------------|
| Rule ID      | DSA029                                                                                        |
| Title        | The `RequiredAttribute` has no impact on a not-nullable value type                             |
| Category     | Bug                                                                                           |
| Severity     | ⚠ Warning                                                                                     |
| Code fix     | ✅                                                                                             |
| Is enabled   | ✅                                                                                             |

## Description

It is a common misunderstanding that the [Required Attribute](https://learn.microsoft.com/en-us/dotnet/api/system.componentmodel.dataannotations.requiredattribute) is somehow able to validate a
not-nullable [value type](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/builtin-types/value-types) property
(`bool`, `int`, `long`, `short`, `byte`, `sbyte`, `ushort`, `uint`, `ulong`, `float`, `double`, `decimal`, `char`, `Guid`, enums, structs, etc.).  
In reality, not-nullable, not-string types are ignored by the [Required Attribute](https://learn.microsoft.com/en-us/dotnet/api/system.componentmodel.dataannotations.requiredattribute), so it doesn't
make any sense to use it in this context.

This rule excludes `DateTime` and `DateTimeOffset`, which are covered by [DSA008](DSA008.md) and [DSA009](DSA009.md) respectively.

## Security implications

### CWE-20: Improper Input Validation

The `[Required]` attribute on a non-nullable value type creates a false sense of security: the developer believes the property is validated, but the attribute is silently ignored at runtime. A value type property always has a value (e.g., `0` for `int`, `false` for `bool`, `Guid.Empty` for `Guid`), so the `[Required]` check never fails. This means invalid or default input passes validation unchallenged, potentially reaching business logic, authorization decisions, or persistence layers.

For example, a `[Required] public int TenantId` property will accept `0` without complaint — if downstream code trusts that the value was validated, this can lead to unauthorized cross-tenant data access or data corruption.

**IEC 62443 mapping**: FR 3 (System Integrity) / SR 3.5 (Input validation) — a silently ineffective validation attribute violates the requirement that all inputs be validated before use in safety- or security-critical operations.

## Fix/Mitigation

Remove the [Required Attribute](https://learn.microsoft.com/en-us/dotnet/api/system.componentmodel.dataannotations.requiredattribute), or make the property nullable.
For numeric types, you may also replace the attribute with a [Range Attribute](https://learn.microsoft.com/en-us/dotnet/api/system.componentmodel.dataannotations.rangeattribute) to enforce that the value is not zero.

## Code fix

Two code fixes are offered, depending on the property type.

**Remove `[Required]`** — available for all matched value types (`bool`, `int`, `long`, `short`, `byte`, `sbyte`, `ushort`, `uint`, `ulong`, `float`, `double`, `decimal`, `char`, `Guid`, enums, structs):

```csharp
// Before
[Required] public bool IsActive { get; set; }

// After
public bool IsActive { get; set; }
```

When other attributes are present on the property, only `[Required]` is removed:

```csharp
// Before
[Description("test")]
[Required] public int Count { get; set; }

// After
[Description("test")]
public int Count { get; set; }
```

**Replace with `[Range(1, T.MaxValue)]`** — available for numeric types only (`byte`, `sbyte`, `short`, `ushort`, `int`, `uint`, `long`, `ulong`, `float`, `double`):

```csharp
// Before
[Required] public int Count { get; set; }

// After
[Range(1, int.MaxValue)] public int Count { get; set; }
```

Each numeric type uses its own keyword in the replacement (e.g. `long.MaxValue`, `byte.MaxValue`, `double.MaxValue`).

When the original `[Required]` attribute is fully qualified, the replacement preserves the fully qualified form:

```csharp
// Before
[System.ComponentModel.DataAnnotations.Required] public int Count { get; set; }

// After
[System.ComponentModel.DataAnnotations.Range(1, int.MaxValue)] public int Count { get; set; }
```

The Range fix is **not offered** for:
- `decimal` (no implicit conversion to `double`, which the `Range` constructor requires)
- `bool`, `char`, `Guid` (a numeric range has no meaningful semantics)
- Enums and structs (not numeric primitives)

## Swagger / OpenAPI consideration

Tools such as [Swashbuckle](https://github.com/domaindrivendev/Swashbuckle.AspNetCore) and [NSwag](https://github.com/RicoSuter/NSwag) read `[Required]` to populate the `required` array in the generated OpenAPI specification. Removing `[Required]` from an API model property may cause that property to no longer appear as required in the schema, affecting API documentation and client SDK generation.

For **numeric types**, the **Replace with `[Range(1, T.MaxValue)]`** code fix resolves this: the `[Range]` attribute is recognized by Swagger generators as a validation constraint, so the property retains its constraints in the OpenAPI spec — and communicates the valid range to API consumers more accurately than `[Required]` ever did.

For **non-numeric types** (`bool`, `char`, `Guid`, enums, structs) where the Range fix is not available, removing `[Required]` will change the generated OpenAPI contract. In this case, consider one of the following alternatives:
- Suppress the diagnostic for that specific property with `#pragma warning disable DSA029`
- Make the property nullable (`bool?`, `Guid?`, `MyEnum?`) so that `[Required]` becomes effective
- Use a Swagger-specific attribute (e.g., Swashbuckle's schema filters) to mark the property as required independently of data-annotation validation
