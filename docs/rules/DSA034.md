# DSA034

Single-type file exceeds maximum line count.

| Property     | Value                                       |
|:-------------|:--------------------------------------------|
| Rule ID      | DSA034                                       |
| Title        | Single-type file exceeds maximum line count  |
| Category     | Code Smells                                  |
| Severity     | ⚠ Warning                                    |
| Refactoring available | ✅                                  |
| Code-review tools available | ✅                            |
| Is enabled   | ✅                                            |

## Rationale

This rule fires when a source file contains exactly one top-level type declaration (class, struct, interface, record) and exceeds a configurable maximum number of lines. Large single-type files often indicate that the type has accumulated too many responsibilities and would benefit from being split into multiple partial-class files.

The maximum line count is configurable via `.editorconfig`. The default threshold is **500** lines.

Unlike DSA033 (which fires for multi-type files), DSA034 targets single-type files and offers partial-class splitting strategies instead of per-type extraction.

## Security implications

Excessively large files increase the risk that security-sensitive code goes unnoticed during code review. When a class spans thousands of lines, reviewers may skim or skip sections, potentially missing vulnerabilities, improper access controls, or unsafe data handling. Splitting into focused partial files improves review coverage.

## Knowledge base

- [MITRE, CWE-1080: Source Code File with Excessive Number of Lines of Code](https://cwe.mitre.org/data/definitions/1080.html) — files with too many lines are harder to understand and maintain
- [Single Responsibility Principle](https://en.wikipedia.org/wiki/Single-responsibility_principle) — each class should have one reason to change
- [DSA033: File exceeds maximum line count (multi-type)](DSA033.md) — companion rule for multi-type files

## Code sample

```csharp
// File: OrderService.cs (exceeds max_lines threshold, single type)
namespace MyApp
{
    public class OrderService
    {
        private readonly IRepository _repo;

        public OrderService(IRepository repo) { _repo = repo; }

        public void ImportOrder() { /* ... */ }
        public void ExportOrder() { /* ... */ }
        public void ArchiveOrder() { /* ... */ }

        public void GrantPermission() { /* ... */ }
        public void RevokePermission() { /* ... */ }

        public void ClearCache() { /* ... */ }
        public void WarmCache() { /* ... */ }

        // ... hundreds more lines
    }
}
```

## Matched patterns (code is NOT OK)

```cs
// A source file containing exactly one top-level type
// whose line count exceeds the configured max_lines threshold.
// Default threshold: 500 lines.
// The diagnostic is reported on the first line of the file.
```

## Not matched patterns (code is OK)

```cs
// Files at or below the threshold do not trigger the diagnostic
// Files with two or more top-level types (covered by DSA033 instead)
// Generated code is excluded (files marked with [GeneratedCode] or auto-generated headers)
// Files whose name matches an excluded file pattern (e.g. *.Designer.cs, *.generated.cs)
// Files where the single type inherits from an excluded base type (e.g. EF Core migrations)
```

## Included Code Fixer and Mitigation

The code fixer offers two actions when the file contains a single top-level type:

### Strategy 1: Split by member visibility

The type is split into partial files grouped by access modifier:

- **`TypeName.Ctors.cs`** — constructors, destructors, `Dispose`/`DisposeAsync`, all fields, all properties. Base types and interfaces are declared only on this file.
- **`TypeName.Public.cs`** — public methods
- **`TypeName.Internal.cs`** — internal methods
- **`TypeName.Protected.cs`** — protected methods
- **`TypeName.Private.cs`** — private methods
- **`TypeName.ProtectedInternal.cs`** — protected internal methods
- **`TypeName.PrivateProtected.cs`** — private protected methods
- **`TypeName.NestedTypeName.cs`** — each nested type gets its own file

### Strategy 2: Split by topic

Members are grouped by recurring words in their PascalCase names:

- **`TypeName.Ctors.cs`** — constructors, destructors, `Dispose`/`DisposeAsync` only (fields and properties are assigned by topic)
- **`TypeName.TopicName.cs`** — members whose names share the same topic word (e.g., `ImportOrder`, `ExportOrder` share the topic "Order")
- **`TypeName.Misc.cs`** — members that don't match any top topic
- **`TypeName.NestedTypeName.cs`** — each nested type gets its own file

Topic detection uses PascalCase word decomposition with a configurable list of excluded stop words (common verbs like Get, Set, Create, etc.). The top N topics by frequency are selected (configurable, default 5). Members matching multiple topics are assigned to the highest-frequency one.

## Included Refactoring (DSR004)

Both splitting strategies are also available as **refactorings** from the lightbulb menu, without requiring the DSA034 diagnostic to fire. This is useful when you want to split a file that is below the threshold or when the diagnostic severity is set to `none`. The refactoring requires the type to have at least 2 members.

## Included tools for code review

The `[DSA034] Add comment for code review` action inserts a structured block comment above the first line. The comment includes the rule ID, enriched category, title with documentation link, and relevant external references.

## Rule configuration

### Severity
In order to change the severity level of this rule, change/add this line in the `.editorconfig` file:

```ini
# DSA034: Single-type file exceeds maximum line count
dotnet_diagnostic.DSA034.severity = error
```

### Maximum line count
The threshold is configurable via `.editorconfig`:

```ini
# Default is 500. Set higher for more lenient checks, lower for stricter.
# Note: this threshold is independent of DSA033's threshold.
dotnet_diagnostic.DSA034.max_lines = 500
```

### Excluded file patterns

Entire source files can be excluded from analysis by file name pattern. Patterns use glob syntax with `*` as a wildcard and are matched case-insensitively against the file name only (not the full path).

The following patterns are excluded by default:

- `*.Designer.cs` — WinForms/WPF designer-generated files
- `*.generated.cs` — tool-generated code
- `*.g.cs` — XAML/source-generator output
- `*.g.i.cs` — incremental generator output

Setting this option **replaces** the entire default list (it does not extend it):

```ini
# Default: *.Designer.cs, *.generated.cs, *.g.cs, *.g.i.cs
# Overrides the entire default list when set.
dotnet_diagnostic.DSA034.excluded_file_patterns = *.Designer.cs, *.generated.cs, *.g.cs, *.g.i.cs
```

### Excluded base types

Files where the single type inherits (directly or indirectly) from any of these base types are excluded. The full inheritance chain is walked, so a class whose grandparent is an excluded type is also excluded.

The following base types are excluded by default:

- `Microsoft.EntityFrameworkCore.Migrations.Migration` — EF Core migration classes (`Up`/`Down` methods are expected to be long)
- `Microsoft.EntityFrameworkCore.Infrastructure.ModelSnapshot` — EF Core model snapshots (auto-generated)
- `System.Data.Entity.Migrations.DbMigration` — EF6 migration classes
- `Microsoft.EntityFrameworkCore.DbContext` — EF Core context classes (`OnModelCreating` can be lengthy)

Setting this option **replaces** the entire default list (it does not extend it):

```ini
# Default: EF Core migrations, model snapshots, and DbContext.
# Overrides the entire default list when set.
dotnet_diagnostic.DSA034.excluded_base_types = Microsoft.EntityFrameworkCore.Migrations.Migration, Microsoft.EntityFrameworkCore.Infrastructure.ModelSnapshot, System.Data.Entity.Migrations.DbMigration, Microsoft.EntityFrameworkCore.DbContext
```

### Maximum topics (topic strategy)
The number of topic groups is configurable:

```ini
# Default is 5. Members not matching any top topic go to Misc.
dotnet_diagnostic.DSA034.max_topics = 5
```

### Excluded topic words
The list of words to ignore during topic detection is configurable:

```ini
# Comma-separated list. Overrides the entire default list.
dotnet_diagnostic.DSA034.excluded_topic_words = Get,Set,Create,Delete,Update,Find,Check
```
